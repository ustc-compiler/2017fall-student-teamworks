泛型编程

泛型编程是指程序设计中对还未指定的类型编写算法，在指定类型后才 实例化的编程模式。泛型编程的好处是可以大量减少重复的代码，提高 程序员的效率。

C语言中的泛型编程

void qsort(void* base, size_t num, size_t size,
           int (*compar)(const void*, const void*));

size指定了每个元素的宽度，传入的compar函数在函数体里 将元素解释成指定的类型，从而正确完成较。

缺点

  • 使用强制类型转换使得类型完全无法检查，使用者甚至不能保证传入了数组
  • compar函数仍然无法泛型

原因

c语言没有提供显式泛型机制，由于弱类型语言的特性，程序员可以随意解释变量 从而能实现类似的效果

C++中的泛型编程

理想的泛型实现

  1. 高通用性/良好的表达能力
  2. 比起手写代码不会带来额外开销
  3. 良好的接口定义

现实的模板

  1. 图灵完全
  2. 编写效率高于普通程序
  3. 肮脏的接口

template <参数列表> 声明

参数列表中的形参可以是类型形参，也可以是非类型形参

例子

enum school {
    ZJU,
    USTC,
    ...
};

template <typename GPAType, enum school m>
void unify_GPA(T x) {}

这里typename T是类型形参

enum school m是非类型形参

f<double, USTC>(4.3);

concepts 101

语法

template <template-parameter-list>
concept concept-name = constraint-expression;

九种限制类型

  • 合取限制
  • 析取限制
  • 断言限制
  • 表达式限制
  • 类型限制
  • 隐式转换限制
  • 参数推导限制
  • 异常限制
  • 参数化限制

合取限制

任意两个限制P和Q的合取是P && Q

当且仅当P、Q都被满足时它才被满足

析取限制

任意两个限制P和Q的析取是P || Q

P、Q中任意一者被满足它都被满足

断言限制

断言限制是个类型为bool的表达式

当且仅当它的值是1的时候才被满足

表达式限制

表达式限制将会替换模板中的参数到表达式中

只有替换成功才会被满足 如：

template <typename T>
concept bool C = requires (T t) { ++t; };

只有++t合法时C才被满足

类型限制

类型限制通过替换模板参数进行类型构造

只有替换后的类型规范时才满足 如：

template <typename T>
concept bool C = requires () { typename T::type; };

隐式转换限制

隐式转换限制E只有能被隐式

转化到T时才能满足限制，如：

template <typename T>
concept bool C =
requires (T a, T b) {
    { a == b } -> bool;
}

C中包含两个限制，分别是表达式限制和隐式转换限制

参数推导限制

参数推导限制只有含占位符的类型T能成

功推出表达式E的类型时才被满足，如：

template <typename T>
concept bool C1 = true;

template <typename T>
concept bool C() {
    return requires(T t) {
        {*t} -> std::pair<C1&, auto>;
    };
}

异常限制

异常限制只有一个表达式E满足noexcept(E)时才被满足

参数化限制

参数化限制声明了一些列参数，叫做限制变量

它们在requires的参数列表中被声明

它的操作数被定义为一系列限制的合取

参数化限制

template <typename T>
concept bool Eq =
requires (T a, T b) { 
    a == b;
    a != b;
};

只有参数类型进行替换后仍然规范

并且操作数都满足时参数化限制才满足

定义concepts

使用concept和requires关键字就可以定义concepts

使用concepts

使用concepts时也需要使用requires语句

这里requires语句用来指定模板参数的限制

使用concepts

template <typename T>
requires Eq<T>
bool f(T t);

它定义了一个模板函数f，模板参数类型T必须满足Eq这个concept所定义的限制才能被实例化成功

缩写形式

template <Eq T>
bool f(T t);

Eq{T} bool f(T t);

template<typename T>
bool f(T t) requires Eq<T>;

bool f(Eq);

将限制翻译到模板

我们尝试用模板来达到一样的功能

基础模板

template <typename ...>
using void_t = void;

template <typename T, typename = void>
struct DIP_concept : std::false_type { };

template <typename T>
struct DIY_concept<T, void_t<填入限制>> : std::true_type { };

合取限制

模板别名P和Q的合取可以直接将它们加入到void_t的参数列表中。 而类_P和_Q的合取可以表示为将std::enable_if<_P::value && _Q::value>::type 加入到void_t的参数列表中

析取限制

模板别名不能表示析取。类_P和_Q的析取可以表示为将

std::enable_if<_P::value || _Q::value>::type加入到void_t的参数列表中

断言限制

将std::enable_if<P>::type加入到

void_t的参数列表中来增加P的断言限制

表达式限制

将decltype(E)加入到void_t

的参数中来增加E的表达式限制

类型限制

将T加入到void_t的参数列表中来增加T的类型限制

隐式转换限制

任一隐式转换限制{ E } -> T可以将decltype(static_cast<T>(E))

加入到void_t的参数列表中来增加类型限制

参数推导限制

由于auto当前的推导功能有限

所以无法支持参数推导限制

异常限制

任一异常限制noexcept(E)可以将std::enable_if<noexcept(E)>::type

加入到void_t的参数列表中来增加异常限制

参数化限制

参数T t可以用static_cast<T>(*(T*)0)来进行表示

这样的缺点是不能有名字，不过能起到等价的效果

Concepts作为模板的接口

使用Concepts设计accumulate

template <class InputIt, class T>
T accumulate(InputIt first, InputIt last, T init);

accumulate的声明

应该如何设计concept？

Accumulable设计

requires(InputIt it, T t) {
    { (*it) + t } -> T;
}

Accumulable使用

Accumulable{InputIt, T}
T accumulate(InputIt first, InputIt last, T init);

Concepts作为占位符

Sortable x = f(y)

auto f(Container) -> Sortable;

使用Concepts设计的优点

  1. 接口易懂
  2. 报错信息更友好

Concepts作为类型设计的辅助

Concepts配合static_assert可以测试类型设计

static_assert<Number<MyType::value_type>>
static_assert<Sortable<MyType>>

总结

concepts的设计契合了C++的设计思想:

  • 为用户提供了良好的接口
  • 语义上一致
  • 不让用户去做重复劳动
  • 零运行时开销
  • 设计简单
